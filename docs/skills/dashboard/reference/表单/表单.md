# 表单（创建/编辑 + 校验）

后台写入数据的标准形状：一份 zod 校验的表单，字段错误就地显示，提交按钮无效即禁用，服务端错误回填。蓝本建在 TanStack Form + zod 的表单系统上（`TextField`/`NumberField`/`SelectField`/`TextareaField` + `FormError` + `SubmitButton`），创建和编辑共用同一份表单。

试这几下：

- 什么都不填直接点**保存** —— 每个字段的错误就地亮出，按钮随即禁用，改到合法才恢复
- 名称填「经典白瓷杯」（列表里已有）再提交 —— 模拟服务端唯一约束，错误**映射回名称字段** + 顶部横幅
- 价格填 20000（客户端范围内）提交 —— 服务端拒绝但归因不到字段，只出**表单级横幅**
- 点列表里的**编辑** —— 表单带值重挂载；保存成功必 toast，新记录立刻出现在列表顶部并标「刚保存」

## 规矩

- **错误分两层**：字段校验错误在 touched 后显示在字段下方；服务端错误走顶部 `FormError` 横幅。demo 再进一步：能归因到字段的服务端错误（唯一约束）映射回字段，归因不到的（套餐上限）留在横幅
- **提交按钮无效或提交中即禁用**：蓝本 `SubmitButton` 订阅 `canSubmit`/`isSubmitting` 自动处理，消费者不手接这套管线；demo 用同一语义自实现
- **创建/编辑共用一份表单，靠 `key` 重挂载重置**：蓝本在弹窗里用 `key={record?.id ?? "new"}`，每次打开都是干净状态；demo 的编辑/新建切换用的正是这一招
- 真实版把 schema 拆两份：**非 coercing 的 `*FormSchema`** 给 `validators.onChange`（输入类型必须等于表单值），**coercing 的 `*InputSchema`** 给服务端边界（防字符串输入）；`NumberField` 清空时发 `undefined`。demo 和蓝本的页面 demo 一样只用一份内联校验
- demo 用 `setTimeout` 模拟服务端往返；真实版 `onSubmit` 里 `await mutateAsync(payload)`，失败走 `setServerError`，成功后失效该资源的查询键 + toast
